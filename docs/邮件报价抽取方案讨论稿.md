# 备选标题：邮件报价抽取统一方案（清洗 + 内容重建 + LLM 抽取）

## 文档说明

本文档用于沉淀当前关于“邮件报价抽取方案”的产品讨论结论，重点记录方向判断、设计原则、处理流程与后续建议，作为后续产品设计、技术方案和实现排期的基础文档。

当前共识：

- 初步清洗是必要的
- 后续信息抽取尽量交给 LLM
- 不希望在产品层面硬拆成多条完全独立的分类流程
- 需要兼容邮件正文、HTML 表格、Excel、PDF、图片等多种报价载体

## 背景问题

业务中会遇到多种模式的报价邮件，包括但不限于：

- 报价凌乱地放在正文
- 报价以表格形式放在正文
- 报价以 `.xlsx` 格式放在附件
- 报价以 `.pdf` 格式放在附件
- 报价是一张图片

这些输入形式差异很大。如果完全依赖规则或正则去分别处理，维护成本高，面对新格式时准确率不稳定。

## 方向判断

产品方向不建议定义为“按文件类型写多个强规则解析器”，而建议定义为：

**先做必要清洗，把邮件及附件内容统一重建成高质量、结构清晰、适合 LLM 理解的中间表示，再由 LLM 完成报价识别和字段抽取，最后用规则校验和人工复核兜底。**

这个方向的核心思想是：

- 轻预处理，不做过重的前置理解
- 不强依赖“五类邮件”分类分流
- 尽量把不同载体的信息统一成一套可读上下文
- 让 LLM 负责语义理解和字段归一
- 用规则负责底线校验而不是主解析

## 总体方案

建议采用一条统一流水线：

1. 轻清洗
2. 内容重建
3. LLM 抽取
4. 结果校验与人工兜底

## 1. 轻清洗

### 目标

只做必要降噪，不做复杂业务判断。目的是让后续内容更适合模型阅读。

### 建议清洗内容

- 邮件签名
- 历史转发头
- 自动免责声明
- 重复引用内容
- HTML 样式、脚本、无意义标签
- 超长空白、重复换行
- 图片或附件的无关元数据

### 原则

- 不要过度裁剪，避免误删报价内容
- 只删除高确定性的噪声
- 保留内容原有顺序和上下文

## 2. 内容重建

### 目标

把来自不同载体的内容，整理成一份统一、可读、可供 LLM 理解的业务材料。

### 为什么需要“内容重建”

如果只是简单清洗后直接把原始内容交给 LLM，会存在这些问题：

- HTML 噪声仍然很多
- OCR 结果可能乱序
- 表格结构容易丢失
- 正文和附件内容关系不清晰
- 模型容易漏读、误读或混淆来源

因此需要引入一个中间层，把邮件内容变成“人类采购专员也能快速读懂”的形式。

### 重建后的推荐结构

建议统一整理成如下结构：

```text
邮件主题:
发件人:
收件时间:

正文摘要:
...

正文中发现的表格:
表格1:
| 列1 | 列2 | 列3 |
| ... | ... | ... |

附件摘要:
附件1: xxx.xlsx
提取到的表格:
| 件号 | 数量 | 单价 | 交期 | ... |

附件2: xxx.pdf
提取到的正文:
...
```

### 设计要求

- 正文、HTML 表格、Excel、PDF、图片内容都尽量转换为统一文本/表格表示
- 保留来源标识，便于后续追溯
- 不要求这一层做最终字段判断
- 只要求这一层把材料“整理好”

## 3. LLM 抽取

### 目标

由 LLM 统一完成报价识别、字段抽取、语义归一和来源关联。

### 建议让 LLM 负责的内容

- 判断哪些内容是真正有效报价
- 从正文、表格、附件中提取关键字段
- 合并正文和附件之间的互补信息
- 识别币种、税态、交期、MOQ、证书等语义
- 处理字段缺失和表达差异
- 统一输出结构化 JSON

### 建议输出字段

- `part_number`
- `product_name`
- `quantity`
- `currency`
- `unit_price`
- `cny_price`
- `lead_time`
- `moq`
- `certificate`
- `shipping_term`
- `supplier_name`
- `quote_status`
- `remarks`
- `source_location`
- `confidence`

### 产品要求

- 必须强制 Schema 输出
- 不允许模型自由输出自然语言结果
- 必须输出来源位置，例如：
  - 来自正文第 3 段
  - 来自附件 1 第 2 个表格第 5 行
  - 来自 PDF 第 2 页

## 4. 结果校验与人工兜底

### 目标

LLM 负责“理解”，规则负责“兜底”。

### 建议校验内容

- 件号格式是否合法
- 价格是否为有效数字
- 币种和价格表达是否一致
- 数量是否异常
- 行级关键字段是否缺失
- 同一件号是否存在多条矛盾报价
- 是否出现“暂无报价”“不能提供”“无库存”等否定语义

### 建议输出策略

- 高置信度：自动入库
- 中置信度：人工确认
- 低置信度：标红审核

## 两段式 LLM 设计建议

为了降低噪声和提高准确率，不建议整封邮件一步抽字段，建议拆成两段：

### 第一段：报价定位

目标：

- 让模型先找出哪些部分可能是有效报价内容
- 输出候选正文片段、候选表格、候选附件内容块

### 第二段：字段抽取

目标：

- 只对已定位的候选内容做结构化字段抽取

### 这样做的好处

- 降低上下文噪声
- 降低漏读概率
- 提高 token 使用效率
- 更利于后续做人工复核和问题定位

## 和传统方案的对比

### 传统方案：按五类载体分别写强规则解析器

优点：

- 对高结构化模板可做到高精度
- 单点性能可控

缺点：

- 维护成本高
- 新格式适配慢
- 非结构化正文处理困难
- 规则会越来越碎，长期不可持续

### 当前建议方案：清洗 + 重建 + LLM + 校验

优点：

- 对格式变化更有弹性
- 更适合复杂真实邮件场景
- 上线速度更快
- 可逐步优化，不必一开始写满所有规则

缺点：

- 模型调用成本更高
- 对中间内容重建质量要求高
- 不能完全跳过校验和人工复核

## 产品模块建议

建议把系统能力拆成以下模块：

- `Cleaner`
  - 去噪、去签名、去转发链、去 HTML 垃圾
- `Rebuilder`
  - 把正文、表格、附件提取结果统一重建成标准材料
- `Locator LLM`
  - 识别真正有报价价值的内容块
- `Extractor LLM`
  - 抽取结构化字段
- `Validator`
  - 做规则校验、异常检测、置信度打分
- `Reviewer UI`
  - 提供人工确认、修改、追溯来源的位置

## 推荐落地顺序

### 第一阶段

- 先做好内容清洗
- 建立统一内容重建层
- 用单个 LLM 输出结构化 JSON
- 增加基础字段校验

### 第二阶段

- 引入“两段式 LLM”
- 增加更细的来源定位
- 增加更稳定的置信度机制

### 第三阶段

- 建人工确认后台
- 支持人工修正
- 支持后续沉淀供应商模板和经验规则

## 当前阶段的一句话结论

当前最推荐的产品方案是：

**先做最小必要清洗，把邮件及附件内容重建成统一、清晰、可读的业务上下文，再由 LLM 统一完成报价识别和字段抽取，最后用规则校验和人工复核兜底。**
