# 邮件报价抽取技术方案

## 1. 背景

本方案对应 `docs/PRD/邮件报价抽取PRD.md`，目标是实现邮件报价的统一抽取、结果校验与反馈闭环能力。

当前技术问题主要包括：

- 输入载体多样，正文、HTML、Excel、PDF、图片结构差异大
- 直接规则解析维护成本高
- 一次性抽取后缺少可持续优化机制

## 2. 方案概述

整体采用统一流水线：

1. 轻清洗
2. 内容重建
3. LLM 抽取
4. 结果校验与人工兜底
5. 反馈沉淀与策略调整

核心思想是：

**先把邮件及附件整理成结构清晰、可追溯的统一上下文，再让大模型完成语义抽取，最后用规则和人工反馈兜底。**

## 3. 输入输出定义

### 3.1 输入

- 邮件主题、发件人、收件时间
- 邮件正文
- HTML 正文
- Excel 附件
- PDF 附件
- 图片附件

### 3.2 输出

结构化报价结果应至少包含：

- `part_number`
- `product_name`
- `quantity`
- `currency`
- `unit_price`
- `cny_price`
- `lead_time`
- `moq`
- `certificate`
- `shipping_term`
- `supplier_name`
- `quote_status`
- `remarks`
- `source_location`
- `confidence`

### 3.3 中间表示

统一重建后的内容建议组织为 block 化结构，例如：

- 邮件基础信息
- 正文摘要
- 正文表格
- 附件摘要
- 附件表格
- OCR 或 PDF 提取结果

每个 block 必须保留来源标识，便于追溯。

## 4. 详细设计

### 4.1 清洗层

职责：

- 去签名
- 去历史转发头
- 去免责声明
- 去 HTML 噪声
- 去重复引用和多余空白

原则：

- 只删除高确定性噪声
- 保留原始顺序
- 不做业务判断

### 4.2 重建层

职责：

- 把正文、表格和附件提取结果整理成统一上下文
- 保留来源位置
- 尽量保持表格结构

设计要求：

- 不能把表格简单拍平成纯自然语言
- 不能丢失页码、sheet 名、table 编号等来源信息
- 目标是“人类采购专员可快速读懂”

### 4.3 LLM 抽取层

职责：

- 识别有效报价内容
- 抽取字段
- 合并正文与附件中的互补信息
- 做语义归一

输出要求：

- 必须结构化输出
- 必须符合固定 Schema
- 必须包含来源位置

### 4.4 校验层

职责：

- 件号合法性校验
- 价格数值校验
- 币种一致性校验
- 数量异常校验
- 缺失字段检查
- 否定语义识别

校验结果用于：

- 自动通过
- 人工复核
- 低置信度标红

### 4.5 反馈沉淀层

职责：

- 记录人工修改
- 记录正确样本
- 记录错误画像
- 记录模板画像
- 记录策略调整依据

## 5. 核心实现

### 5.1 LLM 接入方式

建议采用分层策略：

- 简单邮件：单次 LLM 完成抽取
- 复杂邮件：先定位候选报价块，再抽取字段

这样可以兼顾：

- 成本
- 准确率
- 可追溯性

### 5.2 Prompt 与 Schema

建议采用强制 Schema 输出，不允许自由自然语言结果。

Schema 设计重点：

- 字段必须结构化
- 来源位置必须可追踪
- 置信度必须可计算或可传递

### 5.3 规则与人工兜底

规则负责底线校验，不负责主解析。

人工介入用于：

- 修正低置信度结果
- 处理异常模板
- 反馈正确答案和错误原因

## 6. 异常与容错

### 6.1 输入异常

- 无正文
- 无附件
- OCR 失败
- PDF 解析失败

### 6.2 处理异常

- 重建失败
- LLM 输出不符合 Schema
- 抽取结果缺失关键字段

### 6.3 降级策略

- 单页失败时保留其他 block
- LLM 失败时进入人工队列
- 低质量输入直接提高人工优先级

## 7. 性能与成本

关注点包括：

- 单封邮件处理延迟
- LLM 调用次数
- Token 成本
- 附件解析时间

建议的优化方向：

- 先清洗再重建，减少无效上下文
- 简单场景单次调用，复杂场景才两段式处理
- 历史相似样本优先辅助输入，减少模型搜索成本

## 8. 监控与日志

建议记录以下指标：

- 抽取成功率
- 人工修改率
- 自动通过率
- 低置信度占比
- 错误类型分布
- 模板命中率

日志至少保留：

- 原始输入引用
- 重建后的 block
- LLM 输出
- 校验结果
- 人工修正记录

## 9. 风险与权衡

### 9.1 方案优点

- 对格式变化更有弹性
- 适合真实邮件场景
- 易于逐步增强
- 便于和反馈闭环结合

### 9.2 方案代价

- 模型调用成本更高
- 重建层质量要求高
- 需要人工兜底

### 9.3 备选方案

- 按文件类型拆多套强规则解析器
- 端到端直接抽取，不做中间重建
- 先做专用模型训练后再上线

本项目当前不优先采用这些方案。

## 10. 落地顺序

### 第一阶段

- 完成清洗
- 完成统一重建
- 完成单次 LLM 抽取
- 增加基础校验

### 第二阶段

- 引入两段式 LLM
- 增强来源定位
- 增强置信度机制

### 第三阶段

- 接入人工审核后台
- 建立样本沉淀和模板画像
- 形成策略调节闭环
