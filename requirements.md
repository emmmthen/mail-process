# 航空零件采购比价系统 - 技术需求规格说明书

## 1. 项目概述

### 1.1 业务背景
航空零件采购/销售人员在日常业务中，需要从多个供应商处收集报价数据，并手动整理到 Excel 模板中进行比价。该过程耗时且容易出错。

### 1.2 项目目标
开发一个自动化系统，能够：
- 自动识别和提取供应商邮件中的报价数据
- 支持多种数据格式（表格、文本、图片、PDF）
- 自动生成比价单
- 提供用户友好的交互界面

## 2. 功能需求

### 2.1 邮件数据提取模块

#### 2.1.3 货币识别规则
- 识别货币符号：$、USD、US$ → 美金；￥、CNY、RMB → 人民币
- 默认规则：无货币符号时，根据供应商地区推断
- 汇率换算：支持自定义换算公式（由业务人员配置）
  - 示例公式：人民币单价 = 美金单价 × 汇率 + 附加费用
  - 公式配置项：汇率、附加费用、服务费等
  - 公式可在系统设置界面由业务人员调整

#### 2.1.1 支持的数据源类型

**类型 A：结构化表格邮件**
- 输入：邮件正文中包含格式规范的 HTML 表格
- 处理：解析 HTML 表格，提取结构化数据
- 输出：标准化的报价数据记录

**类型 B：非结构化文本邮件**
- 输入：邮件正文中包含零散的主数据信息
- 处理：使用 NLP/正则表达式提取关键字段
- 输出：标准化的报价数据记录

**类型 C：图片邮件**
- 输入：邮件正文中包含报价图片
- 处理：OCR 识别 + 数据提取
- 输出：标准化的报价数据记录

**类型 D：PDF 附件邮件**
- 输入：邮件附件为 PDF 文件
- 处理：PDF 解析 + 数据提取
- 输出：标准化的报价数据记录

#### 2.1.2 特殊业务规则

**无报价处理规则**
- 识别条件：邮件内容包含"暂无报价"或"No Quote"
- 处理方式：该件号（Part Number）的报价数据置空
- 标记：在系统中标记为"暂无报价"状态

### 2.2 数据存储模块

#### 2.2.1 数据模型
需要存储的核心字段：
- 件号（Part Number，必填）
- 供应商名称
- 美金单价（USD Price）
- 人民币单价（CNY Price，由公式计算得出）
- 货币符号（原始报价币种，用于 NLP 识别）
- 换算公式参数（汇率、附加费用、服务费等）
- 交货期（Lead Time）
- 最小起订量（MOQ，可为空）
- 数据来源（邮件 ID/附件路径）
- 提取时间戳
- 数据状态（有效/无效/待确认）

#### 2.2.2 存储要求
- 支持多轮询价数据的历史记录
- 支持数据的增删改查操作
- 数据可追溯性（保留原始邮件引用）

### 2.3 比价单生成模块

#### 2.3.1 比价单功能
- 按件号聚合所有供应商报价
- 美金单价和人民币单价分列显示
- 对比人民币单价，高亮显示最低人民币单价
- 支持导出为 Excel 格式
- 支持自定义 Excel 模板

#### 2.3.2 展示要求
- 表格形式展示所有报价
- 高亮显示最优价格
- 支持按价格/交货期等条件排序
- 支持筛选特定供应商

### 2.4 用户交互模块

#### 2.4.1 核心交互
- 邮件导入（手动/自动）
- 数据提取进度展示
- 提取结果预览与确认
- 手动修正数据
- 比价单生成与导出
- 汇率换算公式配置（业务人员可调整）

#### 2.4.2 用户界面要求
- 简洁直观的操作界面
- 批量操作支持
- 实时状态反馈

## 3. 技术架构要求

### 3.1 系统架构
```
┌─────────────────┐
│   用户界面层     │  (Web/Desktop 应用)
├─────────────────┤
│   业务逻辑层     │  (数据处理、比价逻辑)
├─────────────────┤
│   数据提取层     │  (邮件解析、OCR、PDF 处理)
├─────────────────┤
│   数据存储层     │  (数据库、文件存储)
└─────────────────┘
```

### 3.2 技术选型建议

#### 前端方案
- **推荐**：Web 应用（React/Vue + TypeScript）
  - 优点：跨平台、易部署、界面美观、支持多用户访问
  - 缺点：需要浏览器环境

#### 后端方案
- **推荐**：Python
  - 丰富的数据处理库（pandas）
  - 成熟的 OCR 方案（pytesseract）
  - PDF 处理库（PyPDF2、pdfplumber）
  - 邮件处理库（imaplib、email）

#### 数据库方案
- **阶段一（单用户）**：SQLite
  - 优点：零配置、无需单独部署、数据存储在本地文件、快速启动项目
  - 缺点：不适合多用户并发访问
- **阶段二（多用户）**：PostgreSQL/MySQL
  - 优点：支持多用户并发、性能更强、适合企业级部署
  - 缺点：需要单独安装和配置数据库服务器

**架构说明**：采用数据库抽象层设计，单用户转多用户时**无需重构业务代码**，仅需：
1. 修改数据库连接配置
2. 执行数据迁移脚本
3. 添加用户认证模块

### 3.3 核心依赖库

| 功能模块 | 推荐库 | 用途 |
|---------|--------|------|
| 邮件处理 | imaplib, email | 邮件读取和解析 |
| HTML 解析 | BeautifulSoup4 | 表格提取 |
| OCR | pytesseract + OpenCV | 图片文字识别 |
| PDF 处理 | pdfplumber, PyPDF2 | PDF 内容提取 |
| 数据处理 | pandas | 数据整理和导出 |
| Excel 操作 | openpyxl, xlsxwriter | Excel 读写 |
| NLP 提取 | 正则表达式 | 货币符号识别（$,￥,€,USD,CNY 等） |

## 4. 非功能性需求

### 4.1 性能要求
- 单封邮件处理时间 < 5 秒（不含 OCR）
- OCR 处理时间 < 15 秒/图片
- PDF 处理时间 < 10 秒/文件
- 支持批量处理（100+ 邮件）

### 4.2 准确性要求
- 结构化数据提取准确率 > 95%
- OCR 识别准确率 > 90%
- 关键字段（件号、价格）需支持人工校验

### 4.3 可维护性
- 模块化设计，便于扩展新的数据源类型
- 完善的日志记录
- 配置化设计（邮件服务器、模板路径等）

### 4.4 安全性
- 敏感数据加密存储
- 邮件服务器凭证安全存储（使用环境变量或加密配置文件）
- 操作日志审计

## 6. 风险和挑战

### 6.1 技术风险
- **OCR 准确率**：图片质量影响识别效果
  - 缓解：提供人工校验界面

- **PDF 格式多样性**：不同供应商 PDF 格式不统一
  - 缓解：支持模板配置

- **邮件格式变化**：邮件客户端差异
  - 缓解：标准化邮件导入格式

### 6.2 业务风险
- **数据准确性**：价格错误可能导致业务损失
  - 缓解：强制人工复核关键数据
